# XDR Code Generator Plan â€” Stellar PHP SDK

## Context

The Stellar PHP SDK has ~309 hand-written XDR type classes in `Soneso/StellarSDK/Xdr/`. When the Stellar protocol changes (new XDR types in `.x` files), these must be updated manually â€” a slow, error-prone process. The iOS and Flutter SDKs solved this with a Ruby-based generator using the `xdrgen` gem. This plan brings the same approach to PHP.

The generator reads canonical `.x` files from the `stellar/stellar-xdr` repository via xdrgen's AST, produces PHP files matching existing code patterns, and adds 100+ new types not currently in the SDK.

Reference implementations:
- iOS SDK: `/Users/chris/projects/Stellar/stellar-ios-mac-sdk/tools/xdr-generator/`
- Flutter SDK: `/Users/chris/projects/Stellar/stellar_flutter_sdk/tools/xdr-generator/`

## Guiding Principles

1. **No breaking changes to user-facing SDK API** unless fixing a genuine bug.
   - Generated field types, method signatures, and class names must match existing hand-written types.
   - Use override files to preserve SDK naming conventions where they differ from XDR canonical names.
2. **Bug fixes are allowed** even if they cause breaking changes.
   - Missing fields, wrong types, incorrect enum values, missing union cases.
   - All bug fixes documented in `bugs.md` with description and breaking change impact.
   - Easy fixes applied directly; uncertain fixes escalated to project owner.
3. **Format before comparing** â€” normalize both generated and hand-written code before diffing to eliminate cosmetic noise (blank lines, extra variables, spacing inconsistencies).
4. **Sequential batch processing** â€” process types in small batches, validate each batch fully before proceeding.
5. **Session continuity** â€” all state persisted to files (`progress.md`, `learnings.md`, `bugs.md`) to survive session loss or context compaction.

## Generated File Template

Every generated PHP file must include:

```php
<?php declare(strict_types=1);

// This file was automatically generated by xdrgen.
// DO NOT EDIT or your changes may be overwritten.

namespace Soneso\StellarSDK\Xdr;
```

- `declare(strict_types=1)` matches existing SDK convention.
- Header comment enables `xdr-clean-generated` detection via `grep`.
- Namespace matches existing `Soneso\StellarSDK\Xdr` namespace used by all XDR files.
- `.gitattributes` marks all generated files for PR diff collapsing:
  ```
  # All XDR files are generated except infrastructure and wrappers
  Soneso/StellarSDK/Xdr/*.php linguist-generated=true
  # Exclude hand-written infrastructure (override back to false)
  Soneso/StellarSDK/Xdr/XdrEncoder.php linguist-generated=false
  Soneso/StellarSDK/Xdr/XdrDecoder.php linguist-generated=false
  Soneso/StellarSDK/Xdr/XdrBuffer.php linguist-generated=false
  ```
  Wrapper files (e.g., `XdrAccountID.php`) are also excluded via explicit `linguist-generated=false` entries. The full exclusion list is maintained alongside BASE_WRAPPER_TYPES + permanent SKIP_TYPES.

## PHP Code Patterns

### Encoding/Decoding Infrastructure

Existing infrastructure (not generated, permanently in SKIP_TYPES):
- `XdrEncoder.php` â€” static utility: `integer32()`, `unsignedInteger64()`, `opaqueFixed()`, `opaqueVariable()`, `string()`, `boolean()`, etc.
- `XdrDecoder.php` â€” **legacy** static utility (superseded by `XdrBuffer` for stateful reads). Retained for backward compatibility; some external callers may still reference it. Not used by generated code.
- `XdrBuffer.php` â€” stateful reader maintaining position during sequential XDR reading. **All generated `decode()` methods use `XdrBuffer`.**

### XDR -> PHP Type Mapping

| XDR Type | PHP Type | Encode Method | Decode Method |
|----------|----------|---------------|---------------|
| int | int | `XdrEncoder::integer32()` | `$xdr->readInteger32()` |
| unsigned int | int | `XdrEncoder::unsignedInteger32()` | `$xdr->readUnsignedInteger32()` |
| hyper | BigInteger | `XdrEncoder::bigInteger64()` | `$xdr->readBigInteger64()` |
| unsigned hyper | int | `XdrEncoder::unsignedInteger64()` | `$xdr->readUnsignedInteger64()` |
| bool | bool | `XdrEncoder::boolean()` | `$xdr->readBoolean()` |
| string | string | `XdrEncoder::string()` | `$xdr->readString()` |
| opaque[N] | string | `XdrEncoder::opaqueFixed()` | `$xdr->readOpaqueFixed(N)` |
| opaque<> | string | `XdrEncoder::opaqueVariable()` | `$xdr->readOpaqueVariable()` |

**Note on 64-bit types:** PHP has no unsigned 64-bit integer type. `unsigned hyper` maps to `int` (signed 64-bit on 64-bit platforms). For fields where the SDK uses `BigInteger` (from `phpseclib3`) instead of `int`, use `FIELD_TYPE_OVERRIDES` to preserve the existing type. The default typedef chain resolution must be documented in `learnings.md` as each case is encountered.

### SDK-Specific Type Mappings
The PHP SDK uses several custom types not directly in the XDR spec. These must be accounted for in `field_overrides.rb` and `type_overrides.rb`:

| SDK Type | Usage | Source |
|----------|-------|--------|
| `BigInteger` | 64-bit signed amounts/balances | `phpseclib3\Math\BigInteger` |
| `XdrDataValueMandatory` | Non-optional opaque data | Hand-written subclass of `XdrDataValue` |

**Phase 2 requirement:** Produce an exhaustive audit of all SDK-specific custom types by scanning existing XDR files for types not in the `.x` AST. Document each one's purpose, which fields use it, and the override needed. This audit goes into `learnings.md` under "SDK-Specific Type Audit".

### PHP Reserved Words

The generator must handle PHP reserved words that could appear as field names or identifiers. PHP 8.1+ reserved words include: `abstract`, `array`, `as`, `break`, `callable`, `case`, `catch`, `class`, `clone`, `const`, `continue`, `declare`, `default`, `do`, `echo`, `else`, `elseif`, `empty`, `enum`, `eval`, `exit`, `extends`, `final`, `finally`, `fn`, `for`, `foreach`, `function`, `global`, `goto`, `if`, `implements`, `include`, `instanceof`, `insteadof`, `interface`, `isset`, `list`, `match`, `namespace`, `new`, `null`, `or`, `print`, `private`, `protected`, `public`, `readonly`, `require`, `return`, `static`, `switch`, `throw`, `trait`, `try`, `unset`, `use`, `var`, `while`, `xor`, `yield`, `true`, `false`, `int`, `float`, `string`, `bool`, `void`, `never`, `mixed`.

Strategy: PHP allows reserved words as property names (`$obj->class`) and constant names (`self::DEFAULT`) in most contexts. However, the override audit (Phase 2) must specifically check for collisions and use `member_overrides.rb` or `field_overrides.rb` to rename any problematic identifiers.

### Generated Type Patterns

**Enum:**
```php
<?php declare(strict_types=1);

// This file was automatically generated by xdrgen.
// DO NOT EDIT or your changes may be overwritten.

namespace Soneso\StellarSDK\Xdr;

class XdrAssetType {
    private int $value;

    const ASSET_TYPE_NATIVE = 0;
    const ASSET_TYPE_CREDIT_ALPHANUM4 = 1;
    const ASSET_TYPE_CREDIT_ALPHANUM12 = 2;
    const ASSET_TYPE_POOL_SHARE = 3;

    public function __construct(int $value) {
        $this->value = $value;
    }

    public function getValue(): int {
        return $this->value;
    }

    public function encode(): string {
        return XdrEncoder::integer32($this->value);
    }

    public static function decode(XdrBuffer $xdr): XdrAssetType {
        $value = $xdr->readInteger32();
        return new XdrAssetType($value);
    }
}
```

**Struct:**
```php
class XdrPrice {
    public int $n;
    public int $d;

    public function __construct(int $n, int $d) {
        $this->n = $n;
        $this->d = $d;
    }

    public function encode(): string {
        $bytes = XdrEncoder::integer32($this->n);
        $bytes .= XdrEncoder::integer32($this->d);
        return $bytes;
    }

    public static function decode(XdrBuffer $xdr): XdrPrice {
        $n = $xdr->readInteger32();
        $d = $xdr->readInteger32();
        return new XdrPrice($n, $d);
    }
}
```

**Union (enum discriminant):**
```php
class XdrAsset {
    protected XdrAssetType $type;
    private ?XdrAssetAlphaNum4 $alphaNum4 = null;
    private ?XdrAssetAlphaNum12 $alphaNum12 = null;

    // constructor, getters/setters

    public function encode(): string {
        $bytes = $this->type->encode();
        switch ($this->type->getValue()) {
            case XdrAssetType::ASSET_TYPE_NATIVE:
                break;
            case XdrAssetType::ASSET_TYPE_CREDIT_ALPHANUM4:
                $bytes .= $this->alphaNum4->encode();
                break;
            // ... more cases
        }
        return $bytes;
    }

    public static function decode(XdrBuffer $xdr): XdrAsset {
        $result = new XdrAsset();
        $result->type = XdrAssetType::decode($xdr);
        switch ($result->type->getValue()) {
            case XdrAssetType::ASSET_TYPE_NATIVE:
                break;
            case XdrAssetType::ASSET_TYPE_CREDIT_ALPHANUM4:
                $result->alphaNum4 = XdrAssetAlphaNum4::decode($xdr);
                break;
            // ... more cases
        }
        return $result;
    }
}
```

**Union (int discriminant):**
Same pattern but discriminant is `int $discriminant` read via `readInteger32()`.

### Nested/Inline Type Handling

XDR unions and structs can contain anonymous nested types (e.g., `TransactionSignaturePayload` has a nested union). The generator produces separate PHP classes for these with composite names: `Xdr{Parent}{Child}.php`. The `name_overrides.rb` file can override these composite names where needed.

### Extension Point Simplification

Many XDR structs have an `ext` field of type `union { 0: void }` (extension points for future protocol versions). These void-only unions are simplified to a single `int` field rather than generating a full union class. An `EXTENSION_POINT_FIELDS` map in `type_overrides.rb` identifies which struct fields should use this simplification. Pattern:
```php
// Instead of a full XdrFooExt union class:
public int $ext = 0;  // reserved extension point
```

### XDR Constants

The `.x` files define constants (`const MASK_ACCOUNT_FLAGS = 0x7`, `const MAX_OPS_PER_TX = 100`, etc.). The generator collects all `AST::Definitions::Const` nodes and emits them into a single `XdrConstants.php` file, following the iOS SDK pattern. If the PHP SDK already defines any of these constants elsewhere, they will be cross-referenced during Phase 2 to avoid duplication.

## Wrapper Strategy (Base + Wrapper Pattern)

### Problem
~25-30 hand-written XDR types contain custom helper methods (factory methods, StrKey conversions, BigInt support, base64 encoding) that the generator cannot produce. These must be preserved without breaking existing SDK API.

### Solution
PHP inheritance-based base+wrapper pattern (same approach as Flutter SDK):

- **Generator produces:** `XdrAccountIDBase.php` with pure encode/decode
- **Hand-written wrapper:** `XdrAccountID extends XdrAccountIDBase` keeps custom methods
- **Rest of SDK** references `XdrAccountID` (no breaking changes)
- **Generator config:** `BASE_WRAPPER_TYPES` list in `type_overrides.rb`

### Self-Referencing Base Types
Some wrapper types like `XdrSCVal` have fields that reference themselves (e.g., `array<XdrSCVal>`). When `XdrSCValBase.php` is generated, these field declarations must reference `XdrSCVal` (the wrapper), not `XdrSCValBase`. PHP autoloading handles this naturally (no circular import issue), but the generator must emit the wrapper class name in field declarations for base wrapper types. A `SELF_REFERENCING_BASE_TYPES` list documents these types explicitly.

### Base64 Methods
~18 types share `toBase64Xdr()`/`fromBase64Xdr()` methods. Rather than introducing a trait (which adds refactoring risk for minimal benefit), the generator emits `toBase64Xdr()` and `fromBase64Xdr()` directly in each generated class as simple inline methods. This is zero risk for backward compatibility.

```php
public function toBase64Xdr(): string {
    return base64_encode($this->encode());
}

public static function fromBase64Xdr(string $xdr): static {
    $decoded = base64_decode($xdr, true);
    return static::decode(new XdrBuffer($decoded));
}
```

### Types Requiring Wrapper Treatment

**High complexity** (extensive custom logic):
- `XdrSCVal` â€” 32 factory methods + toBigInt() + type conversion
- `XdrHostFunction` â€” 8 factory methods with domain logic
- `XdrLedgerKey` â€” 10+ factory methods for different ledger entries
- `XdrTransaction` â€” complex constructor with business defaults
- `XdrMuxedAccount` â€” validation and discriminant logic

**Medium complexity** (standard + custom methods):
- `XdrAccountID` â€” StrKey encoding/decoding integration
- `XdrSCAddress` â€” factory methods + toStrKey() conversion
- `XdrClaimableBalanceID` â€” factory method + getPaddedBalanceIdHex()
- `XdrMuxedAccountMed25519` â€” dual encode/decode (inverted) + getAccountId()
- `XdrContractExecutable`, `XdrContractIDPreimage`
- `XdrDecoratedSignature` â€” toBase64() + getRawSignature()

**Asset types with inheritance** (see "Inheritance-Based Wrappers" below):
- `XdrChangeTrustAsset extends XdrAsset` â€” fromXdrAsset() + liquidity pool support
- `XdrTrustlineAsset extends XdrAsset` â€” fromXdrAsset() + pool share support
- `XdrAllowTrustOperationAsset` â€” fromAlphaNumAssetCode() factory

### Inheritance-Based Wrappers (Permanent SKIP_TYPES)
The asset types above use a different inheritance pattern: they `extend XdrAsset` directly rather than extending a generated base class. The base+wrapper pattern cannot be applied here because the generator would produce `XdrChangeTrustAssetBase` with its own discriminant, but the SDK's `XdrChangeTrustAsset` inherits `XdrAsset`'s discriminant and adds extra arms.

These types must remain in **permanent SKIP_TYPES** (hand-written, not generated):
- `XdrChangeTrustAsset` â€” extends `XdrAsset`, adds `ASSET_TYPE_POOL_SHARE` arm
- `XdrTrustlineAsset` â€” extends `XdrAsset`, adds pool share support
- `XdrAllowTrustOperationAsset` â€” SDK-specific union not directly in `.x` files

If a future refactor makes these types standalone unions (not inheriting from `XdrAsset`), they can be moved out of SKIP_TYPES and into BASE_WRAPPER_TYPES.

### Pre-Generation Preparation
Before starting batch processing, all wrapper types must be prepared:
1. Identify all types needing wrappers (from the list above)
2. For each wrapper type:
   - Create `XdrFooBase.php` by extracting encode/decode from existing `XdrFoo.php`
   - Refactor `XdrFoo.php` to `extends XdrFooBase` with only custom methods
3. Run `composer test` after each wrapper creation
4. Commit each wrapper individually or in small groups (not one big commit) for easy revert
5. Add `BASE_WRAPPER_TYPES` to `type_overrides.rb`

## Override Files

### `name_overrides.rb`
Maps XDR canonical names to PHP class names where the default `Xdr{CamelCase}` convention doesn't match.
- Built by auditing all 309 existing files against xdrgen AST names.
- Must also check for PHP reserved word collisions.
- Example: `"AllowTrustOpAsset" => "XdrAllowTrustOperationAsset"`

### `member_overrides.rb`
Maps enum member names and union arm names where SDK naming differs from XDR.
- Example: enum constants with non-standard prefixes, union case names.

### `field_overrides.rb`
Two sections:
- **FIELD_OVERRIDES**: XDR field name -> PHP property name renames.
- **FIELD_TYPE_OVERRIDES**: Per-class, per-field type overrides to preserve SDK type choices.
  - Example: `"XdrOfferEntry" => { "offerID" => "BigInteger" }` (SDK uses BigInteger where XDR says int64)

### `type_overrides.rb`
- **TYPE_OVERRIDES**: Typedef resolution (e.g., `"XdrTimePoint" => "int"`)
- **BASE_WRAPPER_TYPES**: List of ~25-30 types generating `*Base.php` files
- **SELF_REFERENCING_BASE_TYPES**: Subset of BASE_WRAPPER_TYPES where fields reference the wrapper class
- **SKIP_TYPES**: Initially all types; shrinks per batch (see "Final SKIP_TYPES" below)
- **EXTENSION_POINT_FIELDS**: Map of struct -> field names for void-only union simplification

### Final SKIP_TYPES (Post-Completion)
The permanent SKIP_TYPES list after all batches are complete should contain only:
- `XdrEncoder`, `XdrDecoder`, `XdrBuffer` â€” encoding infrastructure, not XDR types
- Any types defined outside `Soneso/StellarSDK/Xdr/` that the generator cannot place correctly
- Types that require thread-safety or other patterns incompatible with generation (unlikely in PHP)

This list should be documented in the generator's README as the expected final state.

### Note on iOS/Swift-Specific Overrides
The iOS SDK has `LET_TYPES`, `SPECIAL_FIELDS`, `INIT_PARAM_OVERRIDES`, and `INIT_PARAM_ORDER` â€” these are Swift-specific (immutability, computed property defaults, init parameter sugar) and are not needed for PHP. If PHP constructors have parameter order differences from XDR field order, a `CONSTRUCTOR_OVERRIDES` equivalent can be added later if needed.

## Sequential Batch Processing

### Strategy
Start with ALL existing types in SKIP_TYPES. Remove types in small batches (5-60 per batch), validate fully, then proceed. Batch ordering follows complexity progression:

### Batch Order
1. **Simple enums** (5-10 types) â€” validate basic rendering, naming conventions
2. **More enums** (10-20 types) â€” bulk enum processing
3. **Simple typedefs** (5-10 types) â€” int/string/opaque wrappers
4. **Simple structs** (5-10 types) â€” 2-3 field structs, validate field handling
5. **Simple unions** (5-10 types) â€” enum and int discriminants, void arms
6. **Base wrapper types** (5-10 types) â€” validate base+wrapper pattern works
7. **Complex structs** (10-20 types) â€” optional fields, arrays, nested types
8. **Operation types** (20-30 types) â€” operation structs + result codes + results
9. **Soroban/contract types** (20-30 types) â€” complex nested structures
10. **Remaining types** â€” clear SKIP_TYPES progressively
11. **New types** (100+ types) â€” types in `.x` files not currently in SDK

### Per-Batch Cycle

```
 1. Remove batch types from SKIP_TYPES
 2. Run generator (make xdr-generate)
 3. Run compare.sh â€” format both sides, diff, report exact/semantic/mismatch counts
 4. Categorize each difference:
    a. Generator bug       -> fix the generator
    b. Hand-written bug    -> log in bugs.md, fix if straightforward, ask owner if uncertain
    c. Missing wrapper     -> pause, create base+wrapper split, re-generate
 5. Run validate_generated_types.rb (structural validation against .x files)
 6. Run composer test (all 2,601+ unit tests must pass)
 7. Add round-trip encode/decode tests for new types in this batch
 8. Run composer test again (new tests must also pass)
 9. Agent code review for production readiness (see review checklist below)
    - If issues found: fix, re-run tests, re-review (loop until clean)
10. Present batch results to project owner:
    - Types generated, match quality (exact/semantic)
    - Bugs found (with bugs.md references)
    - Breaking changes (if any)
    - Reviewer sign-off status
11. Owner approval -> commit batch
12. Update progress.md, learnings.md
13. Proceed to next batch
```

### Agent Review Checklist
The agent code review at step 9 must verify:
- All fields match `.x` definitions (covered by validation script)
- Encode/decode methods handle all edge cases (null/optional fields, empty arrays)
- No PHP reserved word collisions
- Naming conventions match override files
- No unnecessary type casting or precision loss
- Generated code has correct namespace and strict_types declaration
- Base wrapper types correctly reference wrapper class in self-referencing fields
- No missing `use` imports for external types (BigInteger, etc.)
- Wrapper types that inherit `fromBase64Xdr()` must override `decode()` to return `static` correctly (PHP late static binding: `XdrFooBase::decode()` returns `XdrFooBase`, but `XdrFoo::fromBase64Xdr()` calls `static::decode()` which must return `XdrFoo`). Each wrapper must either override `decode()` or override `fromBase64Xdr()` to cast the result.

### Diff Automation (`compare.sh`)
Script that automates the format-both-sides-then-diff workflow:
```bash
#!/bin/bash
# Usage: ./compare.sh [file_pattern|all]
# 1. Copies generated and original files to temp directory
# 2. Normalizes both sides using sed-based whitespace/formatting rules:
#    - Strip trailing whitespace
#    - Collapse multiple blank lines to one
#    - Normalize brace spacing (e.g., "function (){" -> "function () {")
#    - Normalize visibility keyword spacing
# 3. Diffs normalized versions
# 4. Reports: exact matches, semantic matches, mismatches
#
# NOTE: We use sed-based normalization rather than php-cs-fixer to keep the
# Ruby Docker image sufficient (no PHP toolchain needed). If deeper formatting
# is needed later, compare.sh can optionally invoke php-cs-fixer inside a
# php:8.2-cli Docker container.
```
Added to Makefile as `xdr-compare` target (runs inside Docker). Used in every batch to reduce manual effort.

### Diff Noise Reduction
Hand-written PHP code has inconsistencies (mixed visibility, blank lines, spacing). To reduce diff noise:
- `compare.sh` formats both sides before diffing
- Document cosmetic patterns in `learnings.md` so subsequent batches filter them faster
- Examples of known cosmetic diffs:
  - `encode() : string` vs `encode(): string` (space before colon)
  - Blank lines between property declarations
  - Inline construction vs separate variables in decode()

## Validation Script (`validate_generated_types.rb`)

Adapted from iOS SDK (`/Users/chris/projects/Stellar/stellar-ios-mac-sdk/tools/xdr-generator/test/validate_generated_types.rb`). Parses `.x` files via xdrgen AST and validates each generated PHP file.

**Note:** This is a non-trivial adaptation. The iOS script uses Swift-specific regex patterns (`public struct Foo: XDRCodable`). The PHP version needs PHP-specific patterns for: class declarations (`class XdrFoo`), property declarations (`public ?Type $name`), method signatures (`public function encode(): string`), and constant declarations (`const FOO = N`).

**Note:** The iOS script references `ADDITIONAL_SKIP_TYPES` which appears undefined â€” the PHP version must define this explicitly (even if empty) to avoid runtime errors.

### What It Validates
- **Structs:** field count, field names, field types match `.x` definition
- **Enums:** case count, case names, case values match
- **Unions:** case count, case names, associated types, void vs typed arms match
- **Typedefs:** target type matches (wrapper class, opaque, string, array)

### Why It Matters
- Round-trip tests don't catch missing fields (encode/decode still passes, just silently ignores)
- Catches structural drift when `.x` files are updated in new protocol releases
- Validates new types that have no hand-written code to diff against
- Exit code 1 on any failure â€” suitable for CI integration

### Usage
```bash
cd tools/xdr-generator && bundle exec ruby test/validate_generated_types.rb
# Or via Makefile:
make xdr-validate
```

## Testing Strategy

### Existing Tests (Safety Net)
- 2,601 unit tests via `composer test` (~2 min runtime)
- 23 existing XDR test files with 160 test methods
- Tests exercise XDR types indirectly through operations, transactions, SEP handlers
- Any generated type that breaks the API contract will cause test failures

### New Round-Trip Tests
For each batch of generated types, add encode->decode round-trip tests:
```php
#[Test]
public function testXdrFooRoundTrip(): void {
    $original = new XdrFoo(/* fields */);
    $encoded = $original->encode();
    $decoded = XdrFoo::decode(new XdrBuffer($encoded));
    // Assert all fields match
}
```

### Test Edge Cases
Every round-trip test should cover relevant edge cases:
- All fields at zero/default values
- Optional fields: present and absent
- Variable-length arrays: empty and non-empty
- Enum: first value, last value, negative values
- Union: each arm, including void arms and default arm
- String: empty string
- Opaque: exactly N bytes for fixed, zero and non-zero for variable

### Cross-Type Integration Tests
Add 5-10 integration-level round-trip tests that encode/decode deeply nested type hierarchies (e.g., `XdrTransaction` containing `XdrOperation` containing `XdrAsset`). The existing 2,601 tests provide some of this, but explicit composition tests catch issues the existing tests miss.

### Test Organization
- New test files in `Soneso/StellarSDKTests/Unit/Xdr/` following existing naming convention
- Group by batch or by type category (e.g., `XdrOperationResultRoundTripTest.php`)
- Each new type gets at least one round-trip test

### Test Execution Per Batch
1. `validate_generated_types.rb` â€” structural correctness (fields, cases, types)
2. `composer test` â€” existing tests pass (no API breakage)
3. Add round-trip tests for batch types
4. `composer test` â€” new tests also pass

## Snapshot Tests

### Purpose
Regression testing for the generator itself. Catches unintended changes to generator output when modifying the Ruby code.

### Structure
8-10 reference snapshot files covering key type patterns:
- Simple enum (e.g., `XdrAssetType.php`)
- Complex enum (e.g., `XdrOperationType.php`)
- Simple struct (e.g., `XdrPrice.php`)
- Complex struct with optionals/arrays
- Union with enum discriminant (e.g., `XdrAsset.php`)
- Union with int discriminant
- Typedef (simple, opaque, array)
- Base wrapper type

### Test Flow
```ruby
def test_snapshot_price_xdr
  generated = File.join(@output_dir, "XdrPrice.php")
  snapshot = File.join(SNAPSHOT_DIR, "XdrPrice.php")
  assert_equal File.read(snapshot), File.read(generated)
end
```

### Update Workflow
When generator changes are intentional:
```bash
make xdr-generator-update-snapshots  # regenerate baselines
# Review diff, commit new snapshots
```

	rm -rf xdr/

	docker run --rm -v $(PWD):/wd -w /wd $(RUBY_IMAGE) /bin/bash -c '\
		cd tools/xdr-generator && \

# Update snapshot baselines
xdr-generator-update-snapshots: $(XDR_SRCS)
	docker run --rm -v $(PWD):/wd -w /wd $(RUBY_IMAGE) /bin/bash -c '\
		cd tools/xdr-generator && \

		cd tools/xdr-generator && \

## Breaking Change Strategy

When generator output differs from hand-written code due to a genuine bug fix:
- **Missing fields**: Additive, not breaking â€” document in `bugs.md`
- **Wrong field types** (e.g., `int` -> `BigInteger`): Breaking â€” document in CHANGELOG with migration instructions
- **Renamed constants/members**: Breaking â€” document in CHANGELOG
- **Missing union cases**: Additive for decode, breaking if users switch on cases â€” document in CHANGELOG

All breaking changes are collected during batch processing and presented as a single CHANGELOG entry at merge time.

- `field_overrides.rb` â€” new fields may need type or name overrides
